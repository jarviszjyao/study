# Lambda Internal Processing Flow

This document defines the internal execution lifecycle of the
Chat Orchestrator Lambda.

The Lambda acts as a stateless orchestration engine.

---

## High-Level Flow

API Request
    ↓
Session Load
    ↓
Context Assembly
    ↓
Planner Invocation (LLM)
    ↓
Planner Decision
    ├── Clarification
    ├── Execute Query
    └── Reject
    ↓
Query Execution (if needed)
    ↓
Response Formatting
    ↓
Session Update
    ↓
Return Response

---

## Step Details

### 1. Request Validation
Validate request against:

schemas/chat.request.schema.json

Reject invalid payloads immediately.

---

### 2. Session Loading

Load session from DynamoDB using:

session_id

If not found:
- create new session
- initialize conversation state

---

### 3. Context Assembly

Build LLM context using:

- recent conversation turns
- resolved entities
- schema registry metadata
- previous query context

See:
context-assembly.md

---

### 4. Planner Invocation

Lambda calls LLM planner with:

- system prompt
- assembled context
- user message

Expected output:
QuerySpec OR Clarification instruction.

---

### 5. Decision Routing

Planner output determines execution:

| Planner Result | Action |
|---|---|
| clarification_required | ask user question |
| execute_query | run query |
| unsupported_request | return safe message |

---

### 6. Query Execution

QuerySpec → SQL Builder → Database

SQL is NEVER generated by LLM.

---

### 7. Response Formatting

Convert raw data into:

chat.response.schema.json

Optional visualization included.

---

### 8. Session Persistence

Update:

- conversation history
- resolved filters
- clarification progress

Store back to DynamoDB.

---

## Lambda Time Constraint

All steps are synchronous and designed
to complete within seconds.

Lambda does NOT maintain long-lived reasoning.
